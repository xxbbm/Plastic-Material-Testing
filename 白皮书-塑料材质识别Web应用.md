# 《塑料材质识别》本地 Web 应用 — 白皮书

## 一、项目概述

### 1.1 背景

塑料回收行业长期面临材质识别难题。一线回收工人和贸易商依赖经验判断——闻气味、看火焰、听声音、折弯测试，经验不足的人容易把合金当纯料、把阻燃料当普通料，造成经济损失甚至安全隐患。行业内流传着"黑话"体系（如 PC 叫"磁碳"、POM 叫"赛钢"），但没有系统化的数字工具把这些经验沉淀下来。

### 1.2 产品定位

一款纯前端本地运行的塑料材质识别 Web 工具，不需要安装、不需要后端服务器，浏览器打开就能用。核心通过**加权评分算法**将燃烧/物理测试的特征观察转化为材质判定，同时接入 **DeepSeek AI** 提供智能咨询对话。

### 1.3 目标用户

- 塑料回收站点的一线分拣工人
- 再生料贸易商（需要快速判定货物材质）
- 废塑料回收企业（培训新员工的辅助工具）

## 三、核心功能模块

### 3.1 详细检测（决策核心）

这是整个应用的核心。采用**先分支后打分**的混合架构：

**第一阶段：测试方式选择**
```
用户选择 ──┬── 火烧测试（4题：火星走向→火焰颜色→烟雾→气味）
           └── 物理感官（5题：透明度→敲击声→弯折→表面→重量）
```

**第二阶段：加权评分**

每道题的每个选项都包含一个权重矩阵，覆盖全部 24 种材质：

```
例：气味题 → 选择"香的/不刺鼻"
├── PE: +4  (蜡烛味)
├── PP: +4  (柴油味偏中性)
├── PC: +4  (花果甜香)
├── PMMA: +4 (果香味)
├── PS: +3  (苯乙烯甜味)
├── PET: +3 (酸味)
├── PLA: +3 (焦糊甜味)
├── PBT: +3 (轻微)
├── 铅料/石头料: +1 (无味)
└── ABS/PA/PVC/POM: 0 (这些料的气味不在"香"的范畴)
```

**置信度公式：**

```
gapFactor = (maxScore - runnerUpScore) / maxScore
baseConfidence = maxScore / (answeredCount × 4.5) × 100
finalConfidence = baseConfidence × (0.4 + 0.6 × gapFactor) - skipCount × 10
```

- 第一名与第二名差距大 → 置信度高（特征区分度好）
- 第一名与第二名差距小 → 置信度低（可能是特征重叠的材料）
- 每跳一题扣 10%，连续跳 2 次触发补偿提醒

### 3.2 AI 智能咨询

**追问式对话模式：**

AI 不会在用户第一次描述后直接给出结论，而是作为诊断式对话者：

1. 分析已有特征，判断还缺什么关键信息
2. 每次只问一个最关键问题（优先燃烧特征，区分度最高）
3. 用户说"不是/不确定"，立刻换方向，绝不重复
4. 最多 4 轮强制输出结论，格式固定：

```
【判定结果】
材质：PC/ABS合金 (合金料)
置信度：高
依据：火星散开 + 折痕发白 + 不透明 + 沉闷声
```

**辅助功能：**
- 底部 11 个快捷标签一键发送
- 结果下方的"结果有误？"按钮，点击后引导用户补充特征重判
- 多聊天会话管理：新对话按钮独立会话，旧对话 7 天自动删除

## 四、数据流与存储策略

### 4.1 材质数据流

```
管理员 /admin 编辑材质
        ↓
PUT /api/materials/[id]
        ↓
data/materials.json ← lib/materials-store.ts 写入磁盘
        ↓
前端 GET /api/materials → 检测/百科自动使用最新数据
```

### 4.2 反馈数据流

```
用户提交反馈 → POST /api/feedback
        ↓
data/feedback.json ← 写入
        ↓
管理员 /admin/feedback 查看 → PATCH 标记处理
        ↓
用户 GET /api/feedback?userId=xxx → 看到已处理状态
```

### 4.3 用户身份

首次访问时 `lib/storage.ts` 生成 UUID 存入 localStorage，作为用户标识。反馈请求携带此 ID。

### 4.4 存储分工

| 数据类型 | 存储位置 | 原因 |
|---------|---------|------|
| 材质数据 | `data/materials.json`（服务端文件） | 需管理员后台编辑，所有用户可见 |
| 反馈数据 | `data/feedback.json`（服务端文件） | 需跨浏览器共享到管理后台 |
| 检测历史 | localStorage | 用户私有，不需要共享 |
| 拍照存样 | localStorage（base64，限 200KB/张） | 纯前端功能 |
| 聊天会话 | localStorage（7 天自动过期） | 避免无限增长 |
| 用户 UUID | localStorage | 持久化用户身份 |

## 五、设计决策与权衡

### 5.1 为什么用 Next.js 而不是 Vite

原白皮书指定 Vite + React，但 v0 生成的前端代码基于 Next.js + shadcn/ui。迁移成本高于收益：
- shadcn/ui 在 Next.js 下零配置
- API Routes 提供了足够用的轻量后端（材质/反馈的 JSON 文件读写）
- 不需要额外的 Express/Flask 服务器

### 5.2 为什么用加权评分而不是纯决策树

决策树在最理想的情况下（用户观察准确、无歧义特征）2-3 步即可出结果。但实际情况是：
- 用户可能把火星散开看成聚拢
- 气味感受主观性强（同一种气味有人觉得臭有人觉得香）
- 部分材料特征重叠（如 PC/ABS 和纯 ABS 有多个共同特征）

加权评分通过收集多个特征来"容错"——一个特征判断错了，其他特征加权后仍然倾向正确答案。同时保留了分支选择（火烧 vs 物理）的聚焦感。

### 5.3 为什么气味只分 4 档

精确气味判断（如"辨别甲醛味和盐酸味的区别"）对一线工人不现实。4 档模糊分类降低了使用门槛，同时通过加权覆盖（每档覆盖 6-15 种材料）保持了区分度。

### 5.4 为什么用 JSON 文件而不是数据库

24 种材质 + 反馈数据量极小，JSON 文件读写零延迟、零配置、可 Git 版本管理。管理后台直接改 JSON 文件，`next dev` 热更新即时生效。

### 5.5 AI 追问模式的设计

一问答模式的问题：用户不知道如何描述，给的信息不够。追问模式让 AI 主动引导，模拟一个有经验的师傅问新手的过程——"先烧一下看看火星"→"火星往哪边跑"→"有没有黑烟"→"闻一下什么味"。平均 3-4 轮即可锁定材质。

## 六、安全考量

| 关注点 | 措施 |
|--------|------|
| DeepSeek API Key | 客户端变量 `NEXT_PUBLIC_` 前缀，只在前端网络请求中使用，不与第三方共享 |
| 管理后台口令 | `.env.local` 配置 + httpOnly cookie，24 小时过期 |
| XSS | React 默认转义，无 `dangerouslySetInnerHTML` |
| 用户数据隐私 | 检测历史/拍照存于浏览器 localStorage，不经过服务器 |

### 6.1 安全警告（应用中已实现）

- POM（赛钢）检测到时自动弹出红色全屏安全警告："火焰几乎不可见，剧毒甲醛，立即通风"
- PVC 标注"不可燃烧测试"警告
- 加铅重料标注"含重金属，建议按杂料处理"
- 置信度低于 60% 的结果页显示黄色提示
